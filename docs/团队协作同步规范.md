## 零、新人 5 分钟上手

### 第 1 步：克隆仓库

```bash
git clone https://github.com/lx-1203/KnowAll_Studio.git
cd KnowAll_Studio
```

### 第 2 步：安装 GitHub CLI 并登录

下载：[https://cli.github.com/](https://cli.github.com/)，安装后：

```bash
gh auth login
# 选择 GitHub.com → HTTPS → Login with a web browser → 按提示完成
```

### 第 3 步：切到 dev 分支，拉最新代码

```bash
git checkout dev
git pull origin dev
```

### 第 4 步：创建你的第一个开发分支

```bash
git checkout -b feature/<你的模块>-<功能描述>
# 例如：git checkout -b feature/frontend-login-page
```

### 第 5 步：开始写代码，然后提交

```bash
git add src/xxx.js
git commit -m "feat(frontend): 完成登录页面布局"
git push origin feature/frontend-login-page
```

### 第 6 步：去 GitHub 创建 Pull Request

```bash
gh pr create --base dev --title "feat(frontend): 登录页面"
```

> 到此为止你已经走完一遍完整流程。下面逐章展开细节，遇到问题时查阅对应章节即可。

## 一、核心原则

1. **单一事实源（Single Source of Truth）**  
   所有代码和文档以 GitHub 仓库 `master` 分支为唯一权威。任何"我本地能跑啊"的争论以 GitHub 上的代码为准。

2. **小步提交、高频同步**  
   每人每天至少 `push` 1 次，`pull` 至少 2 次（开工前 + 收工前）。代码在本地放越久，冲突概率越大。

3. **可追溯**  
   每条 commit 必须关联 GitHub Issue，任何一行代码都能追溯到：**谁写的 → 为什么写 → 哪个任务要求的**。

4. **冲突零容忍**  
   任何 merge conflict 必须在本地解决后再 push。禁止 `--force` 污染共享分支（master / dev）。**你的 force 就是别人的代码丢失。**

## 二、GitHub 分支工作流

### 2.1 分支模型

```
master               ← 稳定可发布代码，只接受 PR 合并，禁止直接 push
  └─ dev             ← 日常集成分支，所有功能先合并到这里验证
       └─ feature/<模块>-<功能简述>   ← 个人开发分支
       └─ fix/<模块>-<bug简述>        ← Bug 修复分支
       └─ docs/<内容简述>             ← 文档更新分支
       └─ hotfix/<简述>               ← 紧急线上修复
```

**你永远不应该直接 commit 到 master 或 dev。所有改动都从 dev 拉分支 → 开发 → 提 PR → 合并回 dev。**

### 2.2 分支保护规则（仓库管理员配置）

在 GitHub 仓库 **Settings → Branches → Add branch protection rule** 中设置：

| 保护规则 | master 分支 | dev 分支 |
|----------|:----------:|:--------:|
| Require a pull request before merging | ✅ | ✅ |
| Require approvals（至少 1 人） | ✅ | ✅ |
| Dismiss stale reviews when new commits are pushed | ✅ | ✅ |
| Require status checks to pass before merging | ✅ | ✅ |
| Require conversation resolution before merging | ✅ | — |
| Do not allow bypassing the above settings | ✅ | — |

### 2.3 分支命名规范

| 类型 | 格式 | 好的示例 | 坏的示例 |
|------|------|----------|----------|
| 功能开发 | `feature/<模块>-<描述>` | `feature/backend-upload-file` | `mywork`、`test` |
| Bug 修复 | `fix/<模块>-<描述>` | `fix/frontend-upload-crash` | `bugfix` |
| 文档 | `docs/<描述>` | `docs/api-reference` | `doc1` |
| 紧急热修 | `hotfix/<描述>` | `hotfix/login-500-error` | `urgent` |

> 规则：全小写英文，单词用 `-` 连接。让人一眼看出**谁在什么模块做什么**。

### 2.4 分支完整生命周期

```
① 从 dev 拉出新分支
    ↓
② 本地开发 + 多次小步 commit
    ↓
③ push 到 GitHub
    ↓
④ 在 GitHub 上创建 Pull Request（PR）
    ↓
⑤ 至少 1 人 Code Review + Approve
    ↓
⑥ GitHub PR 页面点 "Squash and merge"
    ↓
⑦ GitHub 自动提示删除远程分支 → 点删除
```

> ⚠️ 合并后**立即**删除远程功能分支，不然三个月后你会看到 50 个没人知道还在不在用的分支。

## 三、GitHub Issue 驱动开发流程

### 3.1 为什么必须先建 Issue？

- ❌ 不建 Issue：你在写什么？同事不知道。同事在写什么？你不知道。→ **撞车。**
- ✅ 先建 Issue：任务可见、可讨论、可分配、可追溯。→ **清晰。**

### 3.2 Issue 模板

在 GitHub Issues 页面点 **New Issue**，按以下格式填写：

```markdown
### 背景
<!-- 为什么要做这个改动？一句话说明 -->

### 目标
- [ ] 具体任务 1
- [ ] 具体任务 2

### 涉及模块
<!-- 前端 / 后端 / Electron / 文档 / 其他 -->

### 预计影响文件
<!-- 大概会改哪些文件，让队友心里有数 -->
- src/xxx.js
- api/xxx.py

### 关联
<!-- Ref: #依赖的Issue编号（如有） -->
```

### 3.3 Issue → 分支 → PR 关联链

```
Issue #42：添加文件上传功能
  ↓
分支：feature/backend-upload-file
  ↓
commit: feat(backend): 实现文件上传接口 (Ref: #42)
commit: feat(frontend): 添加拖拽上传面板 (Ref: #42)
  ↓
PR 描述：Closes #42
  ↓
PR 合并 → Issue #42 自动关闭 ✅
```

**这样任何一个团队成员都能从一行代码反向追溯到：**
- 这行代码在哪个 PR 中被合并的
- 这个 PR 关联哪个 Issue
- 这个 Issue 是谁在什么时候创建的任务

### 3.4 任务分配

- 团队成员在 Issue 下留言 "我接这个" → 负责人自行将 Issue **Assign** 给自己
- 开始开发时，将 Issue 状态移到 **In Progress**
- 避免两个人同时 assign 同一个 Issue（GitHub 上能直接看到谁被 assign）

## 四、每日同步节奏（防冲突关键）

### 4.1 开工三件事（每天必做）

```bash
# ① 切到 dev，拉取所有同事昨晚的更新
git checkout dev
git pull origin dev

# ② 在浏览器看一眼 GitHub Issues / Projects
#    或者用命令行：
gh issue list --assignee @me

# ③ 从最新 dev 创建今天的开发分支
git checkout -b feature/<模块>-<描述>
```

### 4.2 开发中 — 高频提交

```bash
# 每完成一个子任务（1~2 小时）就做一遍：
git add src/xxx.js src/yyy.js    # 指定具体文件，禁止 git add .
git commit -m "feat(<模块>): <简述>"  # 规范见第五章
git pull origin dev               # 拉别人刚合并的更新
# 如有冲突 → 按第六章流程解决
git push origin <当前分支>         # 推到 GitHub，防止本地数据丢失
```

### 4.3 收工四件事（每天必做，缺一不可）

```bash
# ① 暂存所有进度
git add <文件1> <文件2>
git commit -m "<类型>(<模块>): <今日进度简述>"

# ② 把自己的分支推到 GitHub
git push origin <我的分支>

# ③ 如果功能完成，创建 Pull Request
gh pr create --base dev --head <我的分支> --title "<PR标题>"

# ④ 拉取全团队今日更新
git checkout dev
git pull origin dev
```

> ⚠️ **没人应该第二天早上 pull 下来一堆冲突。收工前必须 push 所有代码到 GitHub。如果你本地硬盘坏了，GitHub 是你唯一的备份。**

### 4.4 同步检查清单（贴在显示器旁边）

```
□ 开工时 pull 过 dev 了吗？
□ 今天 push 过至少 1 次了吗？
□ 收工前所有代码都 push 了吗？
□ PR 提了吗（如果功能已做完）？
```

## 五、Commit Message 规范

### 5.1 标准格式

```
<类型>(<范围>): <简短描述>

<详细描述（可选）>

Ref: #<Issue编号>
```

### 5.2 类型速查表

| 类型 | 用途 | 示例场景 |
|------|------|----------|
| `feat` | 新功能 | 添加文件上传接口 |
| `fix` | Bug 修复 | 修复上传超时未重试 |
| `docs` | 文档变更 | 更新 API 文档 |
| `style` | 格式调整 | 修复缩进、去掉分号 |
| `refactor` | 重构（不改功能） | 提取公共上传逻辑 |
| `perf` | 性能优化 | 上传改为分片并发 |
| `test` | 测试相关 | 补充上传模块单元测试 |
| `chore` | 杂项（依赖/构建） | 升级 axios 到 1.x |

### 5.3 好的 commit vs 坏的 commit

```bash
# ✅ 好的 commit
git commit -m "feat(frontend): 添加文件拖拽上传组件

支持拖拽单个/多个文件到上传区域，自动预览和去重。
Ref: #42"

# ✅ 好的 commit
git commit -m "fix(backend): 修复大文件上传时内存溢出

将上传流从 buffer 改为 stream 直写磁盘，1GB 文件不再 OOM。
Ref: #58"

# ❌ 坏的 commit
git commit -m "update"
git commit -m "fix bug"
git commit -m "."
git commit -m "111"
git commit -m "保存一下"
```

> 你的 commit message 是写给3个月后的自己和其他同事看的。**"update" 没有任何信息量。**

### 5.4 PR 描述中的关键字

| 关键字 | 效果 |
|--------|------|
| `Closes #42` | PR 合并后自动关闭 Issue #42 |
| `Fixes #42` | 同上 |
| `Resolves #42` | 同上 |
| `Ref #42` | 关联但不关闭（Issue 仍需手动关闭） |

## 六、冲突解决规范

### 6.1 预防（比解决重要 10 倍）

- **修改前先 pull**：永远基于最新 dev 分支开新分支。
- **避免同时改同一文件**：发现和别人工作重叠 → 先在 Issue 下沟通谁先合并。
- **一个文件控制 300 行内**：文件越大，多人同时改的概率越大。
- **改公共模块前必须打招呼**：修改 `utils/`、`common/`、`api/` 等共享代码，先在 Issue 下 @所有人。

### 6.2 冲突解决完整流程

```bash
# 第 1 步：拉取最新 dev
git checkout dev
git pull origin dev

# 第 2 步：切回自己分支，rebase
git checkout <我的分支>
git rebase dev

# 第 3 步：如果出现冲突，Git 会提示哪些文件有冲突
# 打开对应文件，找到冲突标记：
#
#    <<<<<<< HEAD         ← 自己的改动（当前分支）
#    const MAX = 100
#    =======              ← 分界线
#    const MAX = 200      ← 别人已合并到 dev 的改动
#    >>>>>>> 别人的commit
#
# 手动编辑 → 和同事沟通 → 决定保留哪个（或合并两者）→ 删除标记

# 第 4 步：标记冲突已解决
git add <已解决的文件>

# 第 5 步：继续 rebase
git rebase --continue

# 第 6 步：强制更新自己的远程分支（仅限个人分支！）
git push --force-with-lease origin <我的分支>
```

### 6.3 冲突解决原则

| 原则 | 说明 |
|------|------|
| 沟通优先 | 不确定保留哪个时，在 Issue 下 @相关同事，不要自己猜 |
| 功能对等 | 两人的功能都要保留，调整位置让代码同时包含二者 |
| 你的归你 | 自己那部分代码最了解，负责决定自己代码在冲突中的去留 |
| 不确定就找人 | 看不懂别人的代码逻辑 → 找 ta 问，不要直接删 |

### 6.4 GitHub PR 界面冲突提示

当 PR 页面显示：
> **"This branch has conflicts that must be resolved"**

→ 在本地按 6.2 流程解决后重新 `git push --force-with-lease`，PR 会自动更新状态。

### 6.5 三种 abort 逃跑路线（出问题时的救生艇）

```bash
# rebase 到一半搞不定了，直接恢复到 rebase 前的状态
git rebase --abort

# merge 到一半想反悔
git merge --abort

# 你想回到最后一个 commit 之前的干净状态
git checkout -- <文件名>      # 放弃单个文件的修改
git reset --hard HEAD         # 放弃所有本地修改（⚠️ 无法恢复）
```

## 七、GitHub Pull Request 规范

### 7.1 PR 模板（已内置在仓库 .github/ 目录）

创建 PR 时会自动填充以下模板：

```markdown
### 变更说明
<!-- 简要说明做了什么改动 -->

### 关联 Issue
Closes #

### 变更类型
- [ ] 新功能 (feat)
- [ ] Bug 修复 (fix)
- [ ] 文档更新 (docs)
- [ ] 重构 (refactor)
- [ ] 其他

### 测试情况
- [ ] 本地测试通过
- [ ] 相关单元测试已更新

### 截图/录屏（前端改动需提供）
<!-- 拖拽图片到此处 -->

### Checklist
- [ ] 代码通过 lint 检查
- [ ] 没有调试代码（console.log 等）
- [ ] 没有引入新的依赖（如有请说明理由）
- [ ] commit 历史清晰，已 squash 无意义的 WIP 提交
```

### 7.2 提交 PR 前自查清单

- [ ] 代码通过本地 lint 检查
- [ ] 所有相关测试通过
- [ ] 没有调试用的 `console.log` / `print` / 注释掉的代码
- [ ] 没有引入新的依赖（如需要，在 PR 描述中说明）
- [ ] PR 描述中写了 `Closes #<Issue编号>`
- [ ] PR 标题清晰：`feat(frontend): 文件拖拽上传`
- [ ] PR 只做一件事，没有"顺便改"的无关内容

### 7.3 合并策略选择

在 GitHub PR 界面点击 **Merge pull request** 时有三种选择：

| 方式 | 适用场景 | 对 git log 的影响 |
|------|----------|-------------------|
| **Squash and merge** ✅ 推荐 | 日常功能分支开发 | 多个 commit 压缩为一条，历史干净 |
| Rebase and merge | 想保留每个独立提交 | 保留所有 commit，线性历史 |
| Create a merge commit | 大型 PR，需显式标记合并点 | 产生一条 merge commit |

> 日常开发统一使用 **Squash and merge**。PR 里的 WIP 小提交会被压成一条干净的记录。

## 八、Code Review 规范

### 8.1 Review 者职责

- **Review 不是挑刺，是团队安全网。** 你的目标是帮队友发现遗漏，不是证明你比 ta 厉害。
- Review 每行变更，确认逻辑正确、无明显漏洞。
- 如果你 approve 了，意味着**你也对这段代码负责**。

### 8.2 Review 时限

| PR 类型 | 要求 |
|---------|------|
| 普通 PR | 4 工作小时内完成 Review |
| 紧急热修 PR | 2 小时内完成 Review |
| 大型 PR（>500 行） | 可宽限至 1 个工作日，但需提前说明 |

### 8.3 Review 关注优先级

```
① 逻辑正确性  ← 会不会有 bug？边界条件处理了吗？
② 性能影响    ← 会有性能退化吗？
③ 安全性      ← 有没有注入、泄露风险？
④ 可维护性    ← 三个月后别人能看懂吗？
⑤ 代码风格    ← 交给 ESLint/Prettier，不要浪费 Review 时间在这上面
```

### 8.4 Review 意见规范

| 状态 | 含义 | 何时使用 |
|------|------|----------|
| **Approve** | 代码没问题，可以合并 | 所有 Checklist 通过 |
| **Comment** | 有事说一下，但不阻止合并 | 建议性意见，Optional |
| **Request changes** | 有必须修的问题，不准合 | Bug、逻辑错误、安全风险 |

> Comment 时如果只是建议，加一句 "Optional" 或 "非阻塞"，让 PR 作者知道哪些是必须改的、哪些是建议。

### 8.5 PR 作者被提意见后的响应

1. 看完意见，逐条回复。合理的直接改，不同意的在 Review 线程下讨论。
2. 修改后 push 新 commit，PR 自动更新。
3. 所有 Request changes 解决后，点 **Re-request review** 通知 Reviewer 回来重新看。

## 九、实时同步机制（技术架构）

### 9.1 场景与同步方式

| 场景 | 同步方式 | 触发时机 |
|------|----------|----------|
| 代码文件 | Git push/pull + PR | 手动 |
| 需求/设计文档 | WebSocket 实时协作 | 自动 |
| 文件上传/资源共享 | 后端存储 + WS 通知 | 自动 |
| 实时消息/提醒 | WebSocket / SSE | 自动 |
| 任务进度跟踪 | GitHub Issues + Projects | 实时 |
| Bug 状态 | GitHub Issues | 实时 |

### 9.2 WebSocket 实时同步架构（协作编辑场景）

```
┌─────────┐         WebSocket           ┌──────────────┐        ┌─────────┐
│  成员 A  │ ◄──────────────────────────►│   后端服务    │ ◄─────►│  Redis   │
│ (编辑器) │     wss://server/sync       │  (消息路由)   │        │ (会话状态)│
└─────────┘                             └──────────────┘        └─────────┘
                                                │
┌─────────┐         WebSocket                  │
│  成员 B  │ ◄──────────────────────────────────┘
│ (编辑器) │
└─────────┘
```

**核心规则：**
1. **CRDT / OT 算法**：编辑操作被转为原子操作广播，多个编辑方的改动自动合并，不会出现"后保存的直接覆盖先保存的"。
2. **版本号单调递增**：每次保存生成严格递增的版本号，客户端比对本地版本和远程版本，决定 push 或 pull。
3. **心跳检测**：客户端每 15 秒发 ping，服务端 30 秒没收到则判定离线并通知其他成员。
4. **离线队列**：离线期间的修改缓存在本地，上线后按时间顺序自动重放。

### 9.3 文件上传同步流程

```
成员 A 上传文件 "复习笔记.pdf"
  │
  ▼
后端：保存文件 → 生成 file_id + version=1 → 写入消息队列
  │
  ▼
WebSocket 广播给同一协作空间的所有在线成员
  │
  ├─► 成员 B（在线）：收到通知 → 看到 "A 上传了 复习笔记.pdf"
  ├─► 成员 C（在线）：收到通知 → 自动/手动拉取
  └─► 成员 D（离线）：下次上线时拉取增量更新列表
```

### 9.4 冲突处理策略

| 情况 | 处理方式 |
|------|----------|
| 同名文件上传 | 后上传者文件名自动加版本后缀：`笔记_v2.pdf` |
| 同时编辑同一个文档 | 最后保存为准，但保留全部历史版本可回滚 |
| 两人同时改同一行代码（Git） | 见第六章，手动解决冲突 |

## 十三、常见错误与 FAQ

### 13.1 常见错误速查

| 错误信息 | 原因 | 解决方法 |
|----------|------|----------|
| `fatal: not a git repository` | 在非仓库目录执行 git 命令 | `cd KnowAll_Studio` 进到仓库目录 |
| `error: failed to push some refs` | 远程有你的分支没有的 commit | `git pull origin <分支>` 先拉再推 |
| `rebase in progress` | 上次 rebase 没完成或没 abort | `git rebase --abort` 放弃 rebase |
| `detached HEAD` | 当前指针不在任何分支上 | `git checkout dev` 切回 dev |
| `Please enter a commit message` | commit 时没写 -m 进入了 vim | 按 `i` 输入 → 按 `Esc` → 输入 `:wq` → 回车 |
| `CONFLICT (content): Merge conflict in xxx` | 和别人的代码有冲突 | 按第六章流程解决 |
| `fatal: unable to access ... port 7890` | 代理没启动 | 启动代理软件，或 `git config --global --unset http.proxy` |
| `Everything up-to-date` | 本地没有远程新的 commit | 正常现象，说明已经是最新了 |

### 13.2 FAQ

**Q: 我能不能直接在 master 上改？**  
A: 不能。master 有分支保护，你 push 不上去，而且这违反团队纪律。

**Q: 下班了功能没做完怎么办？**  
A: commit + push 到自己的远程分支（不要提 PR）。代码上了 GitHub 就是安全的，第二天继续。**千万不要不 push 就下班。**

**Q: 同事一直没 Review 我的 PR 怎么办？**  
A: 先在 PR 下 @ta 提醒一次。如果 4 小时还没反应，群里 @ta。如果是紧急 PR，标题加 `[URGENT]`。

**Q: 我的分支已经合并了，本地分支怎么处理？**  
A: `git checkout dev && git pull origin dev && git branch -d <已合并的分支名>`

**Q: 发现同事 commit 里有 secret/key 怎么办？**  
A: 立即私聊 ta 和仓库管理员。Key 一旦在 GitHub 公开仓库中出现，视为泄露，必须立即去对应平台轮换密钥。

**Q: 我不确定该不该发 PR？**  
A: 发。宁可多发一个 PR 被拒，也不要默默合代码。PR 是团队讨论的地方，不是考试。

**Q: 遇到了本文档没覆盖的情况怎么办？**  
A: 在 GitHub 提一个 `docs` 标签的 Issue，描述你遇到的问题。我们讨论后在规范里补充。

## 十四、纪律红线

> 以下规则**零容忍**，违反任何一条都是团队事故。

1. **禁止 `git push --force` 到共享分支（master / dev）**  
   后果：可能直接覆盖删除同事的代码，无法恢复。使用 `--force-with-lease` 仅限个人功能分支。

2. **禁止绕过 Code Review 直接合并到 master**  
   后果：未经审核的代码可能包含 bug、安全漏洞。分支保护已开启，理论上你做不到，但不要尝试。

3. **禁止提交密钥/密码/Token/证书**  
   后果：公开仓库里的密钥 = 送给全世界。一旦发生，立即轮换所有相关密钥。使用 `.env` + `.gitignore`。

4. **禁止 `git add .` 不检查直接 commit**  
   后果：可能提交了 `.env`、`node_modules`、临时文件、大文件。commit 前必须 `git status` 看一眼。

5. **禁止代码在本地超过 24 小时不 push**  
   后果：第二天可能和 3 个人的改动冲突，解决成本翻倍。最坏情况：硬盘损坏 → 代码丢失。

6. **禁止在 PR 中夹杂无关改动**  
   后果：Reviewer 看不懂你这个 PR 到底要做什么。"顺手改了格式化" → 单独提一个 PR。
