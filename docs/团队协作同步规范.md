# 团队协作实时同步文档规范

> **仓库地址**：[https://github.com/lx-1203/KnowAll_Studio](https://github.com/lx-1203/KnowAll_Studio)  
> **负责人**：[@lx-1203](https://github.com/lx-1203)  
> **目标**：确保团队所有成员的代码/文档更新能被互相看到，实现实时同步，减少冲突，提升协作效率。

---

## 一、核心原则

1. **单一事实源（Single Source of Truth）**：所有代码和文档以 GitHub 仓库 `master` 分支为唯一权威。
2. **小步提交、高频同步**：每人每天至少 `push` 一次，`pull` 至少两次（开工前 + 收工前）。
3. **可追溯**：每条 commit 必须关联 GitHub Issue，禁止无意义的 commit message。
4. **冲突零容忍**：任何 merge conflict 必须在本地解决后再 push，禁止 `--force` 污染共享分支。

## 二、GitHub 分支工作流

### 2.1 分支模型

```
master               ← 稳定可发布代码，开启分支保护，只接受 PR 合并，禁止直接 push
  └─ dev             ← 日常集成分支，所有功能先合并到这里验证
       └─ feature/<模块>-<功能简述>   ← 个人开发分支
       └─ fix/<模块>-<bug简述>        ← Bug 修复分支
       └─ docs/<内容简述>             ← 文档更新分支
       └─ hotfix/<简述>               ← 紧急线上修复
```

### 2.2 分支保护规则（仓库管理员配置）

在 GitHub 仓库 **Settings → Branches → Add branch protection rule** 中设置：

| 规则 | master 分支 | dev 分支 |
|------|:----------:|:--------:|
| Require a pull request before merging | ✅ | ✅ |
| Require approvals（至少 1 人） | ✅ | ✅ |
| Dismiss stale reviews when new commits are pushed | ✅ | ✅ |
| Require status checks to pass before merging | ✅ | ✅ |
| Require conversation resolution before merging | ✅ | - |
| Do not allow bypassing the above settings | ✅ | - |

### 2.3 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 功能开发 | `feature/<模块>-<描述>` | `feature/backend-auth-login` |
| Bug 修复 | `fix/<模块>-<描述>` | `fix/frontend-upload-crash` |
| 文档 | `docs/<描述>` | `docs/api-reference` |
| 紧急热修 | `hotfix/<描述>` | `hotfix/login-500-error` |

### 2.4 分支生命周期

```
① 从 dev 拉出新分支
    ↓
② 本地开发 + 多次 commit
    ↓
③ push 到 GitHub（git push origin <分支名>）
    ↓
④ 在 GitHub 上创建 Pull Request（PR）
    ↓
⑤ 至少 1 人 Code Review + approve
    ↓
⑥ 通过 GitHub PR 界面点击 "Merge pull request" → 选 "Squash and merge"
    ↓
⑦ 删除远程分支（PR 合并后 GitHub 会自动提示删除）
```

> ⚠️ **合并后立即删除远程功能分支**，防止过期分支堆积。GitHub 会在 PR 合并后自动提示删除。

## 三、GitHub Issue 驱动开发流程

### 3.1 Issue 模板

所有任务必须先建 Issue，再开分支开发。Issue 格式如下：

```markdown
### 背景
<为什么要做这个改动>

### 目标
- [ ] <具体任务 1>
- [ ] <具体任务 2>

### 涉及模块
- 前端/后端/Electron/文档

### 关联
Ref: #<依赖的 Issue 编号>（如有）
```

### 3.2 Issue → 分支 → PR 关联链

```
GitHub Issue #42
  → 分支名: feature/backend-upload-file
    → 所有 commit 写: feat(backend): xxx (Ref: #42)
      → PR 描述写: Closes #42
        → PR 合并后 Issue 自动关闭
```

这样任何成员都能从 Issue 追溯到每一行代码是谁写的、为什么写。

## 四、每日同步节奏（防冲突关键）

### 4.1 开工必做（Start of Day）

```bash
# 1. 切到 dev，拉取所有同事昨晚的更新
git checkout dev
git pull origin dev

# 2. 浏览 GitHub Issues 页面，确认今日任务
gh issue list --assignee @me

# 3. 从最新 dev 创建开发分支
git checkout -b feature/<模块>-<描述>
```

### 4.2 开发中 — 高频更新

```bash
# 每完成一个子任务（约 1-2 小时）
git add <具体文件>                     # 禁止 git add . 一把梭
git commit -m "feat(<模块>): <简述>"  # 参考第五章节规范
git pull origin dev                    # 拉别人的更新
# 如有冲突 → 本地解决
git push origin <当前分支>             # 推到 GitHub
```

### 4.3 收工必做（End of Day）

```bash
# 1. 暂存当前进度
git add <文件>
git commit -m "<类型>(<模块>): <今日进度简述>"

# 2. 推到自己分支
git push origin <我的分支>

# 3. 如果功能已完成，在 GitHub 上创建 PR
gh pr create --base dev --head <我的分支> --title "<PR标题>"

# 4. 拉取当日所有同事更新到本地 dev
git checkout dev && git pull origin dev
```

> ⚠️ **原则：没人应该第二天早上 pull 下来一堆冲突。收工前必须把当天工作推到 GitHub。**

## 五、Commit Message 规范

### 5.1 格式

```
<类型>(<范围>): <简短描述>

<详细描述（可选）>

Ref: #<Issue编号>
```

### 5.2 类型定义

| 类型 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档变更 |
| `style` | 格式（空格、分号等，不改变逻辑） |
| `refactor` | 重构（不改变功能） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/工具链/依赖变更 |

### 5.3 示例

```bash
git commit -m "feat(frontend): 添加文件拖拽上传组件

支持拖拽单个/多个文件到上传区域，自动预览和去重。

Ref: #42"
```

```bash
git commit -m "fix(backend): 修复大文件上传时内存溢出

将上传流从 buffer 改为 stream 直写磁盘。

Ref: #58"
```

### 5.4 关键引用关键字（写在 PR 描述中）

| 关键字 | 效果 |
|--------|------|
| `Closes #42` | PR 合并后自动关闭 Issue #42 |
| `Fixes #42` | 同上 |
| `Resolves #42` | 同上 |
| `Ref #42` | 关联但不关闭 Issue |

## 六、冲突解决规范

### 6.1 预防策略

- **修改前先 pull**：永远基于最新 dev 分支开新分支。
- **避免同时改同一文件**：发现和别人工作重叠时，先在群里/Issue 下沟通谁先合并。
- **拆小文件**：单个文件控制在 300 行以内，减少多人同改概率。
- **改公共模块前先打招呼**：修改 `utils/`、`common/`、`api/` 等共享代码，必须在 Issue 下告知所有人。

### 6.2 GitHub 冲突解决流程

```bash
# 1. 拉取最新 dev
git checkout dev && git pull origin dev

# 2. 切回自己分支，rebase dev
git checkout <我的分支>
git rebase dev

# 3. 如有冲突，编辑文件解决
#    <<<<<<< HEAD         ← 自己的改动
#    =======              ← 分界线
#    >>>>>>> 别人的commit  ← 远程的改动

# 4. 解决后标记
git add <已解决的文件>

# 5. 继续 rebase
git rebase --continue

# 6. 推送到 GitHub（仅限自己的分支）
git push --force-with-lease origin <我的分支>
```

> ⚠️ **绝对禁止** `git push --force` 到 `master` 或 `dev`。  
> ⚠️ `--force-with-lease` **仅限个人功能分支**使用。

### 6.3 GitHub PR 界面的冲突提示

当 PR 页面显示 "This branch has conflicts that must be resolved" 时，在本地按上述流程解决后重新 push，PR 会自动更新。

## 七、实时同步机制（技术架构）

### 7.1 场景分类

| 场景 | 同步方式 | 延迟 |
|------|----------|------|
| 代码/文档文件 | GitHub push/pull + PR review | 按需（手动触发） |
| 在线协作文档（如需求文档） | WebSocket 实时同步 | < 500ms |
| 文件上传/资源管理 | 后端统一存储 + WebSocket 通知 | 秒级 |
| 即时消息/通知 | WebSocket / SSE | < 200ms |
| 任务状态/进度 | GitHub Issues + Project Board | 实时 |

### 7.2 WebSocket 实时同步方案（用于协作编辑功能）

```
┌─────────┐     WebSocket         ┌──────────────┐     ┌─────────┐
│ 成员 A  │ ◄─────────────────────►│  后端服务    │ ◄──► │ Redis   │
│ (编辑器) │   wss://server/sync   │ (消息路由)   │     │ (状态)  │
└─────────┘                       └──────────────┘     └─────────┘
                                          │
┌─────────┐     WebSocket                │
│ 成员 B  │ ◄────────────────────────────┘
│ (编辑器) │
└─────────┘
```

**核心规则：**
1. **CRDT / OT 算法**：编辑操作转为原子操作广播，避免直接覆盖。
2. **版本号递增**：每次保存生成单调递增版本号，客户端比对后决定 pull/push。
3. **心跳检测**：每 15 秒发送 ping，超时 30 秒判定离线。
4. **离线队列**：离线期间的修改本地缓存，上线后按序重放。

### 7.3 文件上传/下载同步

```
成员 A 上传文件
  → 后端保存 + 生成 file_id + 版本号
    → 写入 "待通知" 消息队列
      → 通过 WebSocket 广播给同一协作空间的所有在线成员
        → 成员 B/C/D 收到通知，客户端自动/手动拉取更新
```

**冲突处理：**
- **同名文件**：后上传者自动追加版本后缀（如 `复习笔记_v2.pdf`）。
- **同时编辑**：以最后保存版本为准，保留历史版本可回滚。

## 八、GitHub Pull Request 规范

### 8.1 PR 模板

在仓库中创建 `.github/PULL_REQUEST_TEMPLATE.md`：

```markdown
### 变更说明
<!-- 简要说明做了什么改动 -->

### 关联 Issue
Closes #

### 变更类型
- [ ] 新功能 (feat)
- [ ] Bug 修复 (fix)
- [ ] 文档更新 (docs)
- [ ] 重构 (refactor)
- [ ] 其他

### 测试情况
- [ ] 本地测试通过
- [ ] 相关单元测试已更新

### 截图/录屏（前端改动需提供）
<!-- 拖拽图片到此处 -->

### Checklist
- [ ] 代码通过 lint 检查
- [ ] 没有调试代码（console.log 等）
- [ ] 没有引入新的依赖（如有请说明理由）
- [ ] commit 历史清晰，已 squash 无意义的 WIP 提交
```

### 8.2 PR 提交前自查清单

- [ ] 代码通过本地 lint 检查
- [ ] 所有相关测试通过
- [ ] commit 历史清晰（squash 掉无意义的 WIP 提交）
- [ ] 没有调试用的 `console.log` / `print` / 注释代码
- [ ] 没有引入新的依赖（如需要，在 PR 描述中说明理由）
- [ ] PR 描述中写了 `Closes #<Issue编号>`

### 8.3 Code Review 标准

- 每个 PR 必须至少 **1 人 approve** 才能合并（由分支保护强制执行）
- Review 必须在 **4 工作小时内** 完成（紧急 PR 2 小时）
- Review 聚焦：**逻辑正确性 > 性能 > 安全 > 风格**
- 代码风格问题交给自动化工具（ESLint/Prettier），不浪费人力
- 使用 GitHub 的 **Review changes → Approve / Request changes / Comment** 三种状态明确表态

### 8.4 合并方式

在 GitHub PR 界面点击 **Merge pull request** 时：

| 方式 | 适用场景 |
|------|----------|
| **Squash and merge** ✅ 推荐 | 功能分支，将多个 commit 压缩为一条干净记录 |
| Rebase and merge | 想保留每个独立 commit 时 |
| Create a merge commit | 大型 PR 需要显式记录合并点时 |

> 日常开发推荐 **Squash and merge**，保持 master/dev 的 commit 历史干净。

## 九、GitHub 通知与沟通

### 9.1 GitHub 自带通知机制

| 场景 | GitHub 自动行为 |
|------|----------------|
| 被 @mention | 通知中心 + 邮件通知 |
| 被指定为 Reviewer | 通知中心 + 邮件通知 |
| PR 有新 commit/comment | 通知所有参与者 |
| Issue 有新回复 | 通知所有参与者 |
| PR 被合并/关闭 | 通知 PR 作者 |

> 建议团队每人 **Watch 仓库**（右上角眼睛图标 → "All Activity"），确保不遗漏任何更新。

### 9.2 必须手动通知的事项

| 事项 | 通知方式 | 时机 |
|------|----------|------|
| 准备合并大改动（>200 行） | Issue 下 @相关人 + 群消息 | 合并前 30 分钟 |
| 修改共享模块（utils/common/api） | PR 中 @相关人 + 群消息 | 提 PR 时 |
| 数据库 schema 变更 | 新建 Issue + @所有人 | 变更前 |
| 发现潜在冲突 | Issue 下留言 | 立即 |
| 挂起/休假 | 群消息 | 提前 1 天 |

### 9.3 GitHub Project Board（可选，用于任务看板）

在仓库中创建 **Projects** 看板，列结构：

```
📋 Todo  →  🚧 In Progress  →  👀 In Review  →  ✅ Done
```

每个 Issue 拖拽到对应列，团队所有人实时看到任务进度。

## 十二、纪律红线

1. **禁止 `git push --force` 到共享分支（master/dev）** — 违反即事故，可能导致同事代码丢失。
2. **禁止绕过 Code Review 直接 push 到 master** — 分支保护已开启，所有变更必须走 PR。
3. **禁止提交密钥/密码/Token** — 使用 `.env` + `.gitignore`，一旦提交密钥视为泄露，立即轮换。
4. **禁止 `git add .` 不加检查** — 必须 `git status` 确认自己提交了什么文件。
5. **禁止长期不 push** — 本地代码超过 24 小时未推送视为风险，第二天可能产生大范围冲突。
6. **禁止在 PR 中夹杂无关改动** — 一个 PR 只做一件事，拒绝"顺手改"。
