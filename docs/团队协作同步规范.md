# 团队协作实时同步文档规范

> 目标：确保团队所有成员的代码/文档更新能被互相看到，实现实时同步，减少冲突，提升协作效率。

---

## 一、核心原则

1. **单一事实源（Single Source of Truth）**：所有代码和文档以远程仓库 `main` 分支为唯一权威。
2. **小步提交、高频同步**：每人每天至少 `push` 一次，`pull` 至少两次（开工前 + 收工前）。
3. **可追溯**：每条 commit 必须能追溯到具体的人和任务，禁止无意义的 commit message。
4. **冲突零容忍**：任何 merge conflict 必须在本地解决后再 push，禁止 `--force` 污染远程。

---

## 二、Git 分支工作流

### 2.1 分支模型

```
main          ← 生产就绪代码，只接受 merge，禁止直接 commit
  └─ dev      ← 集成测试分支，日常合并目标
       └─ feature/<模块>-<功能简述>   ← 个人开发分支
       └─ fix/<模块>-<bug简述>        ← 紧急修复分支
       └─ docs/<内容简述>             ← 文档更新分支
```

### 2.2 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 功能开发 | `feature/<模块>-<描述>` | `feature/backend-auth-login` |
| Bug 修复 | `fix/<模块>-<描述>` | `fix/frontend-upload-crash` |
| 文档 | `docs/<描述>` | `docs/api-reference` |
| 紧急热修 | `hotfix/<版本号>-<描述>` | `hotfix/v1.2.1-login-fail` |
| 发布 | `release/<版本号>` | `release/v1.3.0` |

### 2.3 分支生命周期

```
1. 从 dev 拉出 → 2. 本地开发 commit → 3. push 到远程 → 4. 提 PR/MR → 5. Code Review → 6. 合并回 dev → 7. 删除远程分支
```

合并后立即删除远程功能分支，防止过期分支堆积。

## 三、每日同步节奏（防冲突关键）

### 3.1 开工必做（Start of Day）

```bash
# 1. 切到 dev
git checkout dev

# 2. 拉取远程最新
git pull origin dev

# 3. 从最新 dev 创建当日开发分支（如需要）
git checkout -b feature/<模块>-<描述>
```

### 3.2 开发中 — 高频更新

```bash
# 每隔 1-2 小时或完成一个子任务后
git add <具体文件>            # 禁止 git add . 一把梭
git commit -m "<类型>: <简述>"
git pull origin dev           # 先拉别人的更新
# 如有冲突 → 本地解决 → commit
git push origin <当前分支>
```

### 3.3 收工必做（End of Day）

```bash
# 1. 暂存当前进度
git add <文件> && git commit -m "<类型>: <今日进度简述>"

# 2. 拉取当日所有同事更新
git checkout dev && git pull origin dev

# 3. 将自己的分支 rebase 到最新 dev（可选但推荐）
git checkout <我的分支>
git rebase dev

# 4. 推送所有分支
git push origin <我的分支>
```

> ⚠️ **原则：没人应该第二天早上 pull 下来一堆冲突。收工前必须把当天工作推到远程。**

## 四、Commit Message 规范

### 4.1 格式

```
<类型>(<范围>): <简短描述>

<详细描述（可选）>

Ref: #<issue编号>
```

### 4.2 类型定义

| 类型 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档变更 |
| `style` | 格式（空格、分号等，不改变逻辑） |
| `refactor` | 重构（不改变功能） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/工具链/依赖变更 |

### 4.3 示例

```
feat(frontend): 添加文件拖拽上传组件

支持拖拽单个/多个文件到上传区域，自动预览和去重。

Ref: #42
```

```
fix(backend): 修复大文件上传时内存溢出

将上传流从 buffer 改为 stream 直写磁盘。

Ref: #58
```

## 五、冲突解决规范

### 5.1 预防

- **修改前先 pull**：永远基于最新代码修改。
- **避免同时改同一文件**：发现和别人工作重叠时，先在群里沟通谁先合并。
- **拆小文件**：单个文件控制在 300 行以内，减少多人同改概率。

### 5.2 解决流程

```bash
# 1. 拉取最新 dev
git checkout dev && git pull origin dev

# 2. 切回自己分支，rebase
git checkout <我的分支>
git rebase dev

# 3. 如出现冲突，逐文件解决
#    <<<<<<< HEAD         ← 自己的改动
#    =======              ← 分界线
#    >>>>>>> 别人的commit  ← 远程的改动

# 4. 解决后标记
git add <已解决的文件>

# 5. 继续 rebase
git rebase --continue

# 6. 推送（注意：rebase 后需要 force，但仅限自己的分支）
git push --force-with-lease origin <我的分支>
```

> ⚠️ **绝对禁止** `git push --force` 到 `main` 或 `dev`。`--force-with-lease` 仅用于个人功能分支。

## 六、实时同步机制（技术架构）

### 6.1 场景分类

| 场景 | 同步方式 | 延迟 |
|------|----------|------|
| 代码/文档 | Git push/pull | 按需（手动触发） |
| 在线协作文档（如需求文档） | WebSocket 实时同步 | < 500ms |
| 文件上传/资源管理 | OSS + 版本号通知 | 秒级 |
| 即时消息/通知 | WebSocket / SSE | < 200ms |

### 6.2 WebSocket 实时同步方案（用于协作编辑）

```
┌─────────┐     WebSocket      ┌──────────────┐     ┌─────────┐
│ 成员 A  │ ◄──────────────────► │  后端服务    │ ◄──► │ Redis   │
│ (编辑器) │    ws://server/sync  │ (消息路由)   │     │ (状态)  │
└─────────┘                     └──────────────┘     └─────────┘
                                       │
┌─────────┐     WebSocket              │
│ 成员 B  │ ◄──────────────────────────┘
│ (编辑器) │
└─────────┘
```

**核心规则：**
1. **OT (Operational Transformation) 或 CRDT**：编辑操作转为原子操作广播，避免直接覆盖。
2. **版本号递增**：每次保存生成单调递增版本号，客户端比对后决定 pull/push。
3. **心跳检测**：每 15 秒发送 ping，超时 30 秒判定离线。
4. **离线队列**：离线期间的修改本地缓存，上线后按序重放。

### 6.3 文件上传/下载同步

```
成员 A 上传文件
  → 后端保存 + 生成 file_id + 版本号
    → 写入 "待通知" 消息队列
      → 广播给同一协作空间的所有在线成员
        → 成员 B/C/D 收到通知，客户端拉取更新
```

**冲突处理：**
- 同名文件：后上传者自动追加版本后缀（如 `报告_v2.pdf`）。
- 同时编辑：以最后保存版本为准，但保留历史版本可回滚。

## 七、Code Review 规范

### 7.1 提交 PR/MR 前自查

- [ ] 代码通过本地 lint 检查
- [ ] 所有相关测试通过
- [ ] commit 历史清晰（squash 掉无意义的 WIP 提交）
- [ ] 没有调试用的 `console.log` / `print` / 注释代码
- [ ] 没有引入新的依赖（如需要，在 PR 描述中说明理由）

### 7.2 Review 标准

- 每个 PR 必须至少 **1 人 approve** 才能合并
- Review 必须在 **4 工作小时内** 完成（紧急 PR 2 小时）
- Review 聚焦：逻辑正确性 > 性能 > 安全 > 风格
- 代码风格问题交给自动化工具（ESLint/Prettier），不浪费人力

## 八、通知与沟通

### 8.1 必须通知的事项

| 事项 | 通知渠道 | 时机 |
|------|----------|------|
| 准备合并大改动（>200 行） | 群消息 | 合并前 30 分钟 |
| 修改共享模块（utils/common/api） | 群消息 + PR 中 @相关人 | 提 PR 时 |
| 数据库 schema 变更 | 群消息 + 文档更新 | 变更前 |
| 发现潜在冲突 | 群消息 | 立即 |
| 挂起/休假 | 群消息 | 提前 1 天 |

### 8.2 日报格式（可选）

```
【今日完成】
- feat: 完成文件上传进度条组件 (#42)
- fix: 修复上传超时未重试的问题

【阻塞点】
- 等待后端 #58 合并后才能继续

【明日计划】
- 完成文件批量下载功能
```

## 十、纪律红线

1. **禁止 `git push --force` 到共享分支（main/dev/release）** — 违反即事故。
2. **禁止绕过 Code Review 直接合入 main** — 所有变更必须走 PR。
3. **禁止提交密钥/密码/Token** — 使用 `.env` + `.gitignore`。
4. **禁止 `git add .` 不加检查** — 必须明确自己提交了什么文件。
5. **禁止长期不 push** — 本地代码超过 24 小时未推送视为风险。
