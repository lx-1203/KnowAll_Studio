## 一、实时同步总览

### 1.1 同步场景

| 场景 | 同步方式 | 延迟 | 核心要求 |
|------|----------|------|----------|
| **协作文档编辑** | WebSocket（CRDT/OT） | < 500ms | 多人同时编辑不丢数据、不互相覆盖 |
| **文件上传/下载** | HTTP 分片上传 + WebSocket 广播通知 | 秒级 | 大文件不阻塞，进度实时可见 |
| **消息/通知** | WebSocket / SSE | < 200ms | 低延迟，送达保证 |
| **任务状态变更** | WebSocket 广播 | < 500ms | 同空间所有人即时看到 |
| **Git 代码同步** | Git push/pull + PR | 手动 | 见附录简要规范 |

### 1.2 核心原则

1. **谁改谁推送**：任何成员的数据变更（编辑、上传、删除），由**发起方客户端**向服务端 push，服务端广播给同空间其他成员。
2. **增量同步**：只传输变更部分（diff / delta），不传全量数据。
3. **版本为准**：所有数据带单调递增版本号，客户端发现版本落后则拉取更新，发现版本冲突则走 OT/CRDT 合并。
4. **离线可恢复**：断线期间的修改本地缓存，重连后自动重放。

## 二、实时文档协作（WebSocket 同步）

### 2.1 架构

```
┌──────────┐  WebSocket   ┌─────────────────┐   Pub/Sub   ┌──────────┐
│  成员 A   │◄────────────►│   后端同步服务    │◄──────────►│  Redis    │
│ (编辑器)  │  wss://...   │  (Collaboration  │            │ (消息广播) │
└──────────┘              │   Server)       │            └──────────┘
                          │                 │
┌──────────┐  WebSocket   │  房间管理        │
│  成员 B   │◄────────────►│  操作转换(OT)    │
│ (编辑器)  │              │  版本控制        │
└──────────┘              └─────────────────┘
```

### 2.2 通信协议

#### 连接建立

```javascript
// 客户端连接
const ws = new WebSocket('wss://server/sync?doc_id=123&user_id=xxx&token=xxx');

// 服务端验证 token → 加入文档房间 → 返回当前文档状态
```

#### 消息格式（统一 JSON 协议）

```json
{
  "type": "operation",          // 消息类型
  "doc_id": "doc_123",          // 文档 ID
  "user_id": "u_01",            // 操作者
  "user_name": "张三",           // 操作者名称（显示用）
  "version": 42,                // 基于哪个版本的操作
  "timestamp": 1719561600000,   // 操作时间戳（毫秒）
  "data": { ... }               // 具体操作载荷
}
```

#### 消息类型定义

| type | 方向 | 说明 | data 载荷 |
|------|------|------|-----------|
| `operation` | C→S→C | 编辑操作 | 见 2.3 操作类型 |
| `ack` | S→C | 服务端确认，返回新版本号 | `{ "new_version": 43 }` |
| `sync_full` | S→C | 全量同步（首次连接时） | `{ "content": {...}, "version": 43 }` |
| `sync_diff` | S→C | 增量同步（版本落后时） | `[{ "op": "insert", ... }, ...]` |
| `cursor` | C→S→C | 光标位置广播 | `{ "row": 10, "col": 5 }` |
| `presence` | S→C | 在线状态变更 | `{ "online_users": [...] }` |
| `heartbeat` | C→S | 心跳 | `{}` |
| `error` | S→C | 错误 | `{ "code": "VERSION_CONFLICT", "msg": "..." }` |

### 2.3 编辑操作类型（OT 操作）

采用 **JSON0** 风格的 OT 操作格式：

#### 文本编辑操作

```json
// 插入文本（位置 10 处插入 "hello"）
{
  "type": "operation",
  "operation": "insert",
  "path": ["content", "paragraphs", 3, "text"],
  "position": 10,
  "value": "hello"
}

// 删除文本（位置 5~8 的 3 个字符）
{
  "type": "operation",
  "operation": "delete",
  "path": ["content", "paragraphs", 3, "text"],
  "position": 5,
  "length": 3
}

// 属性修改（第 3 段加粗）
{
  "type": "operation",
  "operation": "set_attribute",
  "path": ["content", "paragraphs", 3],
  "attribute": "bold",
  "value": true
}
```

#### 结构化操作（列表、表格、任务）

```json
// 列表新增项
{ "operation": "list_insert", "list_id": "tasks", "index": 2, "item": { ... } }

// 列表删除项
{ "operation": "list_delete", "list_id": "tasks", "index": 2 }

// 列表移动项
{ "operation": "list_move", "list_id": "tasks", "from": 2, "to": 4 }

// 字段更新
{ "operation": "field_update", "item_id": "task_42", "field": "status", "value": "done" }
```

### 2.4 OT 操作转换流程

```
        服务端当前版本: V=42
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
  A 发操作        B 发操作        C 发操作
  (based V=42)   (based V=42)   (based V=42)
    │               │               │
    ▼               ▼               ▼
  服务端 OT 变换（按到达顺序）：
  
  ① A 先到 → 直接应用 → 版本变为 43
  ② B 到达 → 基于 V=42，但当前已是 V=43
           → OT 变换 B' = transform(B, A)
           → 以 B' 应用到 V=43 → 版本变为 44
  ③ C 到达 → 基于 V=42，当前已是 V=44
           → OT 变换 C' = transform(C, [A, B'])
           → 以 C' 应用到 V=44 → 版本变为 45
    │
    ▼
  广播给所有客户端：{ type: "operation", ..., version: 45 }
```

**核心逻辑（伪代码）：**

```javascript
function handleOperation(clientOp, currentDoc) {
  // 1. 如果客户端版本落后，做 OT 变换
  let transformedOp = clientOp;
  const opsToTransform = getOpsSince(clientOp.version); // 服务端已应用的后续操作
  
  for (const appliedOp of opsToTransform) {
    // 双向变换：同时变换 clientOp 和 appliedOp
    const [opA, opB] = OT.transform(transformedOp, appliedOp);
    transformedOp = opA;  // 变换后的 clientOp 应用到当前状态
    // appliedOp 已在文档中，不需要更新（但如果要支持 undo 则需要 opB）
  }
  
  // 2. 应用变换后的操作
  currentDoc.apply(transformedOp);
  currentDoc.version++;
  
  // 3. 广播给房间内其他客户端
  broadcast(room, {
    type: "operation",
    ...transformedOp,
    version: currentDoc.version,
    user_id: clientOp.user_id
  });
  
  // 4. 回 ack 给发送者
  sendAck(clientOp.user_id, { new_version: currentDoc.version });
}
```

### 2.5 客户端处理流程

```javascript
class RealtimeDocClient {
  constructor(docId, userId) {
    this.docId = docId;
    this.userId = userId;
    this.localVersion = 0;     // 本地已同步的版本号
    this.pendingOps = [];      // 待发送队列
    this.offlineQueue = [];    // 离线操作队列
  }

  // 用户编辑 → 生成操作 → 发送
  onUserEdit(operation) {
    const op = {
      type: "operation",
      doc_id: this.docId,
      user_id: this.userId,
      version: this.localVersion,
      timestamp: Date.now(),
      data: operation
    };
    
    // 乐观应用（本地立即生效，不等服务端确认）
    this.applyLocal(op);
    this.send(op);
  }

  // 收到服务端广播（别人的操作或自己的确认）
  onServerMessage(msg) {
    switch (msg.type) {
      case "ack":
        // 自己的操作被确认 → 更新本地版本
        this.localVersion = msg.data.new_version;
        break;
        
      case "operation":
        if (msg.user_id === this.userId) return; // 自己的操作忽略
        // 别人的操作 → 应用到本地编辑器
        this.applyRemote(msg);
        this.localVersion = msg.version;
        break;
        
      case "sync_full":
        // 首次连接 → 全量替换
        this.setContent(msg.data.content);
        this.localVersion = msg.data.version;
        break;
        
      case "sync_diff":
        // 断线重连 → 增量追赶
        for (const op of msg.data.ops) {
          this.applyRemote(op);
        }
        this.localVersion = msg.data.version;
        break;
        
      case "error":
        if (msg.data.code === "VERSION_CONFLICT") {
          // 严重冲突 → 请求全量同步
          this.requestFullSync();
        }
        break;
    }
  }

  send(op) {
    if (this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(JSON.stringify(op));
    } else {
      this.offlineQueue.push(op);  // 离线 → 进队列
    }
  }
}
```

### 3.5 客户端文件同步模块

```javascript
class FileSyncClient {
  constructor(spaceId, userId) {
    this.spaceId = spaceId;
    this.userId = userId;
    this.fileList = [];          // 本地文件列表缓存
  }

  // ========== 上传 ==========

  async uploadFile(file, parentId = null) {
    // ① 初始化上传
    const initResp = await fetch('/api/upload/init', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        filename: file.name,
        file_size: file.size,
        mime_type: file.type,
        space_id: this.spaceId,
        parent_id: parentId
      })
    });
    const { upload_id, chunk_size, total_chunks } = (await initResp.json()).data;

    // ② 分片上传（带进度回调）
    for (let i = 0; i < total_chunks; i++) {
      const start = i * chunk_size;
      const chunk = file.slice(start, start + chunk_size);
      await fetch(`/api/upload/${upload_id}/chunk/${i}`, {
        method: 'PUT',
        headers: { 'Content-Type': 'application/octet-stream' },
        body: chunk
      });
      this.onProgress({ chunk: i + 1, total: total_chunks });
    }

    // ③ 完成上传
    const completeResp = await fetch(`/api/upload/${upload_id}/complete`, {
      method: 'POST'
    });
    const fileInfo = (await completeResp.json()).data;

    // ④ 更新本地列表（等待 WebSocket 广播确认）
    this.onFileUploaded(fileInfo);
    return fileInfo;
  }

  // ========== 下载 ==========

  async downloadFile(fileId) {
    const resp = await fetch(`/api/download/${fileId}`);
    if (resp.status === 302) {
      window.open(resp.headers.get('Location'), '_blank');
    } else {
      const blob = await resp.blob();
      const disposition = resp.headers.get('Content-Disposition');
      const filename = disposition?.match(/filename="(.+)"/)?.[1] || 'download';
      this.saveBlob(blob, filename);
    }
  }

  saveBlob(blob, filename) {
    const url = URL.createObjectURL(blob);
    const a = document.createElement('a');
    a.href = url;
    a.download = filename;
    a.click();
    URL.revokeObjectURL(url);
  }

  // ========== WebSocket 事件处理 ==========

  onWsMessage(msg) {
    switch (msg.type) {
      case 'file_uploaded':
        // 队友上传了文件 → 更新文件列表 UI
        this.fileList.push(msg.data);
        this.onFileUploaded(msg.data);
        break;

      case 'file_deleted':
        // 队友删除了文件 → 从列表移除
        this.fileList = this.fileList.filter(f => f.file_id !== msg.data.file_id);
        this.onFileDeleted(msg.data);
        break;

      case 'file_updated':
        // 队友更新了文件 → 更新列表信息
        const file = this.fileList.find(f => f.file_id === msg.data.file_id);
        if (file) {
          file.version = msg.data.new_version;
          file.size = msg.data.size;
        }
        this.onFileUpdated(msg.data);
        break;
    }
  }
}
```

## 四、消息通知实时推送

│                                          │

### 4.3 推送可靠性保证

| 机制 | 说明 |
|------|------|
| **Seq 序列号** | 每条消息带单调递增 seq，客户端检测 gap 则补拉 |
| **ACK 确认** | 关键消息（文件上传完成等）客户端返回 ack，服务端未收到则重推 |
| **消息持久化** | 用户离线期间的消息写入 Redis list，上线后批量推送 |
| **去重** | 客户端按 msg_id 去重，防止重复推送 |

## 五、冲突处理与版本管理

### 5.1 版本号规则

```
每个协作资源（文档/文件夹/任务列表）独立维护版本号
版本号单调递增，格式：{resource_type}:{resource_id}:{version_num}
示例：doc:123:v42

每次成功写入 → version += 1
客户端操作时必须附带 base_version
```

### 5.2 冲突场景与处理

| 场景 | 触发条件 | 处理方式 |
|------|----------|----------|
| **编辑冲突** | 两人基于同一版本修改同一位置 | OT 变换，自动合并（见 2.4） |
| **文件同名上传** | B 上传了和 A 同名的文件 | 自动加版本后缀：`笔记_v2.pdf`，两文件共存 |
| **文件覆盖更新** | B 对 A 刚上传的文件发起"更新" | 版本号校验：B 必须基于最新版本。基于旧版本的更新被拒绝，提示先刷新 |
| **同时删除** | A 和 B 同时删同一文件 | 先到达服务端的删除生效，后到达的返回 "already deleted" |
| **版本严重分叉** | 离线很久后上线，版本落后太多 | 放弃增量追赶，触发全量同步（sync_full） |

## 六、离线与断线重连

### 6.1 心跳与断线检测

```
客户端：每 15 秒发送 ❤ ping
服务端：超过 30 秒未收到 ping → 判定离线
       → 广播 presence 更新给同空间其他人
       → 将该用户标记为 "offline"，开始缓存消息

客户端：WebSocket onclose 触发
       → 3 秒后首次重连
       → 指数退避：3s → 6s → 12s → 24s → ... → 最大 60s
       → 重连成功后比较本地版本号与服务端版本号
```

### 6.2 重连同步流程

```
客户端重连成功
  │
  ├─ 发送：{ type: "reconnect", doc_id: "xxx", local_version: 40 }
  │
  ▼
服务端比对：
  local_version(40) vs server_version(45)
  │
  ├─ 40 == 45 → 无需同步，继续正常通信
  ├─ 40 < 45 且差距 ≤ 50  → 发送增量 ops（sync_diff，v41~v45 的 5 个操作）
  └─ 40 < 45 且差距 > 50   → 发送全量快照（sync_full）
```

### 6.3 离线操作队列

```javascript
// 客户端离线期间
class OfflineQueue {
  constructor() {
    this.queue = [];            // 按操作顺序存储
    this.maxQueueSize = 500;    // 最大缓存 500 个操作，超出则合并
  }

  push(op) {
    // 尝试合并相邻同类型操作（如连续文本插入合并为一条）
    const last = this.queue[this.queue.length - 1];
    if (last && this.canMerge(last, op)) {
      this.queue[this.queue.length - 1] = this.merge(last, op);
    } else {
      this.queue.push(op);
    }
  }

  // 重连后批量重放
  async replay(sendFn) {
    while (this.queue.length > 0) {
      const op = this.queue.shift();
      await sendFn(op);
    }
  }

  canMerge(a, b) {
    // 同用户、同文档、同类型的连续 insert 可合并
    return a.type === 'insert' && b.type === 'insert' &&
           a.doc_id === b.doc_id &&
           a.path.join('.') === b.path.join('.') &&
           a.position + a.value.length === b.position;
  }

  merge(a, b) {
    return { ...a, value: a.value + b.value };
  }
}
```

## 附：Git 协作简要规范

> Git 是代码同步的基础，以下是精简要点，完整规范见仓库 `docs/` 目录。

### 分支模型

```
master（稳定，禁止直接 push）
  └─ dev（日常集成）
       └─ feature/<模块>-<描述>  ← 你的开发分支
```

### 每日节奏

```bash
# 开工：拉最新 dev
git checkout dev && git pull origin dev

# 开发中：小步提交，推 GitHub
git add <具体文件>
git commit -m "feat(<模块>): <简述>"
git push origin <我的分支>

# 收工：必须 push
git push origin <我的分支>
```

### 核心规则

1. 所有改动走 PR，需要至少 1 人 Approve
2. **禁止 `git push --force` 到 master 或 dev**
3. **禁止代码在本地超过 24 小时不 push**
4. Commit 格式：`feat/fix/docs(<模块>): <简述>`
5. 每天开工 pull、收工 push
